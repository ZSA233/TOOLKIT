# AGENTS 入口规范

本文件是整个仓库的公共底线规范。进入具体子项目目录后，除遵守本文件外，还必须同时遵守该目录下的 `AGENTS.MD`。

当前子项目补充规范：

- `services/mtu_tuner/AGENTS.MD`

如果一次改动同时落在多个子项目中，则对应子规范同时生效。

AGENTS 文本新增与修改要求：

- 新增或修改任意 `AGENTS.md` / `AGENTS.MD` 规则时，必须从“完全没有当前任务上下文的读者”视角编写，确保第一次读到该规范的 AI agent 也能直接理解规则、判断条件和预期动作。
- 规范文本禁止依赖“这次改动里的某几个文件名、链路名、缩写、口头约定”才能成立；如果需要举例，例子也必须是通用、可替换、可脱离当前任务成立的。

## 1. 仓库级原则

- 优先保证代码和文档的可读性、可扩展性、可维护性，避免为了局部速度引入长期复杂度。
- 已有基础能力、adapter、client、repo、生成链路、公共组件或工具模块能够承载需求时，默认优先复用或补强它，不要在业务侧旁路再实现一套相似逻辑。
- 如果现有基础能力缺失，但明显具备复用价值，应优先考虑把能力补到基础层，再由业务复用。
- 如果补强基础层会显著扩大改动面、影响既有调用方或存在多种合理演进路径，应先和用户确认，再决定是否允许业务侧局部方案。
- 只有在确认公共层不适合承载、且用户接受局部方案时，才允许在业务模块内做限定范围的补充实现；同时应在代码或文档中说明未复用基础层的原因。

## 2. 排错与修复规则

- 必须区分“猜测性修复”和“已确认修复”，不要把试探性改动包装成最终结论。
- 猜测性修复必须保持局部、可回退、可删除，不要为一个未确认问题一次叠多层防御式补丁。
- 不要为了让测试通过、压住偶发失败或减少噪音，直接改动核心公共层、底层组件或生产默认行为；只有在已确认该处本身就是实际逻辑错误时，才允许这样修。
- 如果某个修复主要是为测试稳定性服务，但会改变核心层的阻塞特性、时序、默认配置或公共契约，且你无法明确判断这是不是产品缺陷，必须先询问用户；明显局部且合理的修复不需要事事请示。
- 非显而易见的最终修复，必须在代码附近补简短注释，说明为什么这样改、在防什么问题。
- 排错优先通过简化结构、收敛状态、缩小影响面定位问题，而不是继续叠条件分支掩盖症状。
- 临时日志、调试分支、排查保护逻辑在问题确认后必须明确保留还是删除，不能长期残留。

## 3. 结构与耦合控制

- 文件过大、职责混杂、或改动时需要在多个不相关区域反复跳转时，应主动拆分。
- 拆分必须按职责边界或领域边界进行，不按行数机械切块。
- 文件变多后必须用目录或包组织，入口文件保持薄，只做装配、编排和导出。
- 测试代码遵守相同规则，不要长期堆成单个巨型测试文件。
- 结构优化不仅为了人类维护，也为了降低 AI agent 的搜索、理解和替换成本。

## 4. Go 代码拆分与命名

- 不要把多个入口流程塞进同一个大文件；一个文件只承载一个主主题，优先按入口流程或功能主题拆分。
- Go 文件名应表达“这是谁的哪条链路”，优先使用主题后缀分组，而不是 `helper`、`util`、`impl`、`misc` 这类模糊命名。
- 新增文件前先判断调用方范围：如果当前只服务一个入口，默认优先并入该入口文件或该入口前缀文件；只有已经被两个及以上稳定入口共同使用时，才抽成共享主题文件。
- 禁止为了“未来可能复用”提前抽共享文件；先按当前真实调用方收口，未来出现第二个稳定调用方后再重构提取。
- 推荐优先按入口流程拆文件，例如 `*_get.go`、`*_update.go`、`*_save.go`、`*_import.go`；其次再按通用能力拆分为 `*_normalize.go`、`*_validate.go`、`*_mapper.go`、`*_snapshot.go`。
- 入口函数应优先放在对应主题文件顶部；reviewer 进入文件后，应能在几秒内找到入口并看懂主流程。
- 入口专属的小函数，函数名应统一使用入口前缀归组，例如 `UpdateRuleCenter`、`updateRuleCenterBuildPatch`、`updateRuleCenterPersist`。
- 跨多个入口复用的函数，不要继续挂在某个入口文件底部，应按通用主题单独落到对应文件。
- 如果一个文件里同时出现“单入口私有函数”和“跨入口共享函数”，必须拆开，禁止混放。
- 文件名必须匹配当前最窄且真实的职责边界，不要让一个按某条链路、某个领域或某类对象命名的文件，长期承载别的主题的通用能力。
- 拆文件的目标是提升可导航性，不只是压缩行数；不要为了减少单文件行数，把强相关链路拆得四分五裂。
- Go 文件设计默认优先满足这三个目标：
  - 5 秒内找到入口
  - 15 秒内看懂主流程
  - 能从文件名和函数名直接判断归属
- 一句原则：函数可以小，但文件必须清晰；命名必须让人一眼知道“这是哪个入口、哪条链路、哪一层逻辑”。

## 5. Monorepo 结构

- `services/`：业务子项目目录。一个子项目可以同时包含多个服务、模块和协议适配。
- `libs/`：跨服务复用的通用库与第三方协议封装，不应绑定单一业务语义。
- `protocols/`：跨服务协议与契约工作区目录。
- `protocols/grpc/protos/`：跨服务 gRPC 协议源；协议调整必须先改这里，再运行生成器同步到消费方。

## 6. 生成代码与源文件边界

- 自动生成文件禁止手改。包括但不限于：
  - Go views 下的 `gen_*`
  - 前端 API client 下的 `gen_*`
  - protobuf 生成产物
- 需要调整生成结果时，必须回到对应源文件修改：
  - HTTP 接口改 blueprint / 配置源
  - gRPC 接口改 `protocols/grpc/protos/`
  - 路由或前端生成物改其真实源文件，而不是直接修改生成产物

## 7. 测试与验证

- 默认优先执行快速验证，不把慢测试作为日常回归默认路径。
- 慢测试、依赖真实网络/浏览器/外部环境的测试必须显式可跳过。
- 优先使用仓库已有命令入口，例如根 `makefile`、子项目脚本、项目内约定的测试命令，而不是临时拼装一套新的执行方式。
- 测试替身、mock、替换点应尽量命中真实归属模块或真实拥有者，而不是偶然可达的转发层、包根薄壳或临时兼容层。
- 所有仅用于测试的支撑代码都必须放在 `_test.go` 文件中，不允许普通 `.go` 文件长期承载测试专用 stub、fixture 或构造函数。
- 测试支撑文件名也必须表达主题归属，优先使用 `<theme>_fixtures_test.go`、`<entry>_stubs_test.go` 这类命名。

## 8. 依赖显式化

- 入口模块、包根、聚合文件保持薄，不做隐藏中转层。
- 新增模块时，依赖应在当前类、函数或结构体附近可见，不依赖“宿主碰巧提供”。
- 默认优先显式组合，不鼓励隐式共享依赖或跨层偷取上下文。
- 如果没有明确兼容性要求，不为内部旧引用习惯保留魔法转发层。

## 9. 跨服务协议与生成

- gRPC 协议源统一放在 `protocols/grpc/protos/`。
- Go 或 Python 侧的 protobuf / gRPC 产物只通过 `protocols/grpc` 下的 `api-gen` 链路刷新，禁止手改。
- 如果变更会同时影响多个子项目，实施时需要一起检查受影响的服务端、客户端、测试和文档。

## 10. 修改前自检

- 本次改动是否只在真正需要的层级实现，没有绕开现有基础设施重复造轮子。
- 这次修复是猜测性修复还是已确认修复，是否保留了足够的判断依据。
- 是否为了排错叠加了本可避免的复杂度，或遗留了未清理的临时排查代码。
- 当前文件或测试是否已经大到值得拆分，入口是否仍然保持薄。
- Go 文件命名和拆分是否能让 reviewer 快速定位入口、判断链路归属，而不是继续依赖阅读上下文猜测。
- 是否误改了任何自动生成文件。
- 是否同步修改了真正的源文件、生成配置或协议源。
- 如果进入了某个子项目目录，是否同时遵守了该目录下的 `AGENTS.MD`。
